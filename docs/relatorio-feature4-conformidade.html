Escopo: HTTP Basic + autorizacao por URL/HttpMethod + seguranca por padrao

1) Resumo da entrega

Foi implementada a camada de seguranca com Spring Security para autenticar usuarios via HTTP Basic e autorizar acesso por roles (`ADMIN` e `USER`) em dois contextos distintos de endpoints, seguindo o principio do menor privilegio e o comportamento de falha segura por padrao.

3) Trechos de codigo (configuracao de seguranca)

3.1 SecurityFilterChain com autenticacao e autorizacao

@Bean
SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
    http
        .csrf(csrf -> csrf.disable())
        .authorizeHttpRequests(auth -> auth
            .requestMatchers(HttpMethod.GET, "/api/v1/livros/enriquecidos/**").hasRole("ADMIN")
            .requestMatchers(HttpMethod.POST, "/api/v1/livros/**").hasRole("ADMIN")
            .requestMatchers(HttpMethod.PUT, "/api/v1/livros/**").hasRole("ADMIN")
            .requestMatchers(HttpMethod.PATCH, "/api/v1/livros/**").hasRole("ADMIN")
            .requestMatchers(HttpMethod.DELETE, "/api/v1/livros/**").hasRole("ADMIN")
            .requestMatchers(HttpMethod.GET, "/api/v1/livros/**").hasAnyRole("ADMIN", "USER")
            .anyRequest().authenticated())
        .httpBasic(Customizer.withDefaults());

    return http.build();
}

3.2 Usuarios em memoria com senha codificada

@Bean
UserDetailsService userDetailsService(PasswordEncoder passwordEncoder) {
    UserDetails admin = User.withUsername("admin")
        .password(passwordEncoder.encode("admin123"))
        .roles("ADMIN")
        .build();

    UserDetails user = User.withUsername("user")
        .password(passwordEncoder.encode("user123"))
        .roles("USER")
        .build();

    return new InMemoryUserDetailsManager(admin, user);
}

@Bean
PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder();
}

4) Evidencias de testes de seguranca

Classe de teste: SecurityConfigWebTest com validacao explicita de cenarios de sucesso e falha (401 sem autenticacao, 403 por role inadequada e acessos permitidos por role correta).

4.1 Cenarios cobertos

  1. Sem autenticacao em GET /api/v1/livros retorna 401.
  2. USER em GET /api/v1/livros retorna 200.
  3. USER em POST /api/v1/livros retorna 403.
  4. ADMIN em POST /api/v1/livros retorna 201.
  5. USER em GET /api/v1/livros/enriquecidos retorna 403.
  6. ADMIN em GET /api/v1/livros/enriquecidos retorna 200.
  7. Sem autenticacao em endpoint nao mapeado retorna 401 (seguranca por padrao).

5) Principios de seguranca aplicados

  • Menor privilegio: USER so possui leitura de catalogo; escrita e contexto enriquecido ficam com ADMIN.
  • Seguranca por padrao: rotas nao autorizadas explicitamente continuam protegidas por autenticacao.
  • Falha segura: ausencia de credencial gera 401; credencial sem permissao gera 403.

6) Conclusao

A Feature 4 foi implementada com aderencia aos requisitos do PDF, com autenticacao HTTP Basic, autorizacao por roles em dois contextos, regras por HttpMethod, seguranca por padrao e evidencias de testes automatizados em GREEN.
